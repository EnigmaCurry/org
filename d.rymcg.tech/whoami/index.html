Whoami

Whoami is a very simple application, but you can learn a lot from it. It should be the very first application you install after Traefik. You will see later on that all of the applications provided by d.rymcg.tech are configured and managed similar to this one. Use this as a learning experience. If you can accomplish gettting whoami installed, you’ll be able to install anything.

What is Whoami?

Whoami is a web application useful for testing that Traefik is working as expected. Its sole purpose is to show a web page that prints the HTTP headers that were sent to it. This can be useful to verify that the application is capable of serving requests, and that all of the configuration is correct.

Default config file (.env-dist)

Every application provided by d.rymcg.tech, including whoami, has a default configuration file named .env-dist. This config file is used as a template, which is copied whenever you configure a new instance of the application. The .env-dist file should never be edited, as it is always the default configuration. Each instance has its own copy, and it is in the copy that you have the opportunity to change those defaults.

Configure Whoami

To configure whoami, run:

d make whoami config

This process copies the default config file (.env-dist) and creates a new config file, specifically for this instance of the application: (.env_prod_default).

The Whoami env file

Every application instance has a unique config file, and the config target creates it. The first thing you see when running d make whoami config is the name of the environment (config) file that it creates:

(stdout)
Configuring environment file: .env_prod_default

All of the whoami .env files are stored in the whoami directory itself, which is in ~/git/vendor/enigmacurry/d.rymcg.tech/whoami (by default). The naming of the file (.env_prod_default) tells us two things:

  • prod is the name of the active Docker context it was created for.
  • default is the name of the instance of whoami (default is the name of the default instance, but if the instance was named foo the file would be named .env_prod_foo instead.).

You can have multiple config files, each one is for a separate Docker context and application instance.

WHOAMI_TRAEFIK_HOST

The first question the config asks for is WHOAMI_TRAEFIK_HOST which is the fully qualified domain name that the whoami app should use for its URL:

(stdout)
WHOAMI_TRAEFIK_HOST: Enter the whoami domain name (eg. whoami.example.com)

: whoami.prod.rymcg.tech

The default name uses the ROOT_DOMAIN variable you set as part of the main d.rymcg.tech config, which is also named after our current Docker context (prod). Realistically, the WHOAMI_TRAEFIK_HOST may be set to any valid domain name, you just need to setup the DNS for it (to point to the IP address of the Docker host).

Sentry authorization

The next question is about sentry authorization, which configures what type of authentication you want to use, and what clients to allow in through the door.

(stdout)
? Do you want to enable sentry authentication in front of this app (effectively making the entire site private)?
> No
  Yes, with HTTP Basic Authentication
  Yes, with Oauth2
  Yes, with Mutual TLS (mTLS)

You can choose any of the choices, and they will configure your application differently:

  • If you select No, then sentry authorization will be turned off. Any client will be able to access the application.
  • If you select Yes, with HTTP Basic Authentication, the application will require all clients to enter a username/password into a dialog presented by the web browser. Client who enter an incorrect username or password will not be able to view the page.
  • If you select Yes, with OAuth2, the application will require all clients to authenticate with another OAuth2 compatible application, which may be a self-hosted Forgejo instance, or it can be an external service like GitHub. Access is granted only to those users who are listed in the corresponding Traefik authorization group that the application is configured for.
  • If you select Yes, with mTLS, the application will require all clients to authenticate with a client mTLS certificate. Access is granted only to those certificate names that are listed in the application’s config.

Edit the config file by hand if necessary

Once the config script has finished, the config file may be inspected to verify valid settings.

[bash]: Run this on your workstation:
d make whoami config-edit

This will automatically open your default text editor to the whoami config file for the current context/instance.

You can also open the file manually, the path is ~/git/vendor/enigmacurry/d.rymcg.tech/whoami/.env_{CONTEXT}_{INSTANCE}.

Install whoami

Once the configuration has been verified, you can install the application:

[bash]: Run this on your workstation:
d make whoami install

Open whoami in your web browser

Once installed, the application should be ready to view in your web browser:

[bash]: Run this on your workstation:
d make whoami open

This will automatically open your default web browser to the URL of the installed whoami application. If you want to do so manually, just go to the same URL as you configured for WHOAMI_TRAEFIK_HOST.

View the logs

It may be necessary to inspect the applicaiton logs, which you can do so as follows:

[bash]: Run this on your workstation:
d make whoami logs
