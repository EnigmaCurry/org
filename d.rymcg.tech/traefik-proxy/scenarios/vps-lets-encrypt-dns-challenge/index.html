Let's Encrypt DNS challenge

This scenario covers a Traefik configuration for a public or private server. This uses Let’s Encrypt to sign TLS certificates, which are valid for use in most web browsers. This scenario will use the DNS-01 challenge type.

Info

This scenario is good for public and private websites and services.

All Let’s Encrypt challenges leak domain names to public CT logs, however, with DNS-01 challenge we can use wildcards, so that only the root domain is leaked, not the exact hostnames.

Warning

This scenario offers a much more powerful feature set, but it requires storing a security sensitive API token, to control the active DNS server of your domain. You must take special steps to store this token only on a secure workstation and and Docker host.

ACME DNS-01 challenge type

ACME challenges are a type of procedure that authenticates your Traefik instance to the Let’s Encrypt API, proving that you are the owner (controller) of your domain name system (DNS). This is important, because it prevents other actors from requesting certificates for your domains.

The “DNS challenge” (DNS-01), is one of two supported ACME Challenge types. DNS-01 has some advantanges, and disadvantages.

  • DNS-01 challenge works for servers that are behind restrictive firewalls. No open ports are necessary.
  • DNS-01 challenge allows wildcard certificate domains (*.example.com), which means less work when you want to deploy a new application, as long as it uses a valid sub-domain of the existing certificate, you won’t need to reconfigure traefik each time.
  • All Let’s Encrypt challenge types participate in public Certificate Transparency logs. Therefore, any domain name you create a certificate for is publicly “leaked” via CT logs. However, if you only use a wildcard domain (*.example.com) in your certificate, then only the root domain name (example.com) will be leaked, and the bots that are reading the CT logs would have to guess your exact sub-domains, in order to crawl them.

This option will create a new username on the Docker host, called traefik. This is necessary to reserve a unique UID for traefik to run as on the system. Run this first before anything else.

For this tutorial, you will use the default entrypoints, so you don’t need to configure anything.

  • web (TCP port 80, but only used as redirect to websecure)

Configure Let’s Encrypt environment

(stdout)
? Which LE environment do you want to use?
> Production (recommended!)
  Staging (untrusted / testing)
Tip

Always choose the Production environment, unless you really know what you’re doing. Production is the only environment that produces valid (trusted) TLS certificates.

Choose DNS-01 ACME challenge

(stdout)
? Which type of ACME challenge should be used?
  TLS-ALPN-01 (default for public servers, easy, but no wildcard certs)
> DNS-01 (requires API key, but good behind firewalls, and allows wildcard certs)
Tip

For this install, choose DNS-01.

The challenge type will let you use wildcard DNS records, and also works behind a firewall, but it requires storing a security token for your DNS API.

Skip: Configure ACME email address

You don’t have to provide your email address, but if you do, Let’s Encrypt can email you about configuration issues, like certitficates about to expire.

Enter ACME DNS provider config

(stdout)
Find the provider code of your supported DNS provider here:
https://go-acme.github.io/lego/dns/#dns-providers

TRAEFIK_ACME_DNS_PROVIDER: Enter the LEGO code for your DNS Provider (eg. digitalocean)

: digitalocean
Tip

You may use any supported DNS provider of LEGO. This example uses the DigitalOcean provider. If you’re using a different platform, you need to look up the supported DNS provider code in the LEGO docs, and type its name. The code is always one word, no spaces.

[bash]: Run this on your workstation:
You need to enter the NAMES of these variables that your provider needs.
You can enter up to 5 variable names and/or leave them blank
TRAEFIK_ACME_DNS_VARNAME_1: Enter the 1st DNS provider variable name (eg. DO_AUTH_TOKEN)

: DO_AUTH_TOKEN

TRAEFIK_ACME_DNS_VARNAME_2: Enter the 2nd DNS provider variable name (or leave blank)

:
Info

Each supported DNS provider requires certain variables to be set, in order to pass the API token. In the LEGO docs for DigitalOcean it lists the variable name that it needs. This question is asking for the literal name of the variables specific to the DNS platform, which for DigitalOcean is: DO_AUTH_TOKEN. If your provider needs more than one variable set, you can enter them on additional lines, but DigitalOcean only needs one variable set, so it leaves the second line blank.

Enter ACME DNS API token

Next it will ask you to enter the values for the variable names you just set. This is where you put your actual API token of your DNS platform.

[bash]: Run this on your workstation:
Now to enter the values for the custom DNS API variables:
DO_AUTH_TOKEN: Enter the value for DO_AUTH_TOKEN (eg. your-actual-digitalocean-token-here)

: dop_xxxxxx

Configure TLS certificates and domains

d.rymcg.tech uses explicit certificate requests configured centrally, on the traefik project:

(stdout)
? Configure Traefik TLS certificates
> Manage all certificates.
  Create a new certificate.
  Done / Go back

Manage all certificates

This will show you all the certificate requests that have been defined and allow you to manage each one. It will be blank to start out with.

Create a new certificate

Use this to define all the certificates you need for all your applications.

Each certificate needs a main name (CN), which should be the main domain name of the certificate.

Enter the main domain (CN) for this certificate
: prod.example.com
Info

Each certificate may also contain several other domain names, known as SANs (Subject Alternative Names). You can use this to list as many additional domain names that you want to be listed on the same certificate. In the special case of DNS-01, you are allowed to use a wildcard domain (*.example.com) for either the main CN or the SANS.

Now enter additional domains (SANS), one per line:
Enter a secondary domain (enter blank to skip)
: *.prod.example.com
Tip

Make sure that each domain name that you entered on the certificate has a valid DNS A record, pointing to your Docker server’s IP address, including any wildcard domain.

The next chapter will require the domain whoami.prod.example.com, and this will automatically be covered by the wildcard domain *.prod.example.com.

Skip: Configure middleware (including auth)

No middleware is needed for this scenario.

Install Traefik

To install Traefik, you can simply choose the Reinstall Traefik option in the menu, or you can run that step all by itself from the command line at any time:

[bash]: Run this on your workstation:
d make traefik install

You should always reinstall Traefik after changing any configuration setting.
